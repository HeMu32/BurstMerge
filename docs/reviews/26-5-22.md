# 2026-05-22 Review Notes

 ## A. 关键错误行为

 ### 1. 普通 TIFF 被误分类为 RAW

 - 相关文件
   - `libburstmerge/src/core/pipeline.cpp:47-57, 65-71, 75-94`

 - 问题描述
	 - `IsRawExtension()` 将 `.tif` / `.tiff` 视为 RAW 扩展名。
	 - `IsImageExtension()` 同时也把它们当普通图片。
	 - `ClassifyInputs()` 先判断 `IsRawExtension(ext)`，因此所有 TIFF 被归类为 RAW，而不是 RGB。

 - 影响
	 - 普通 TIFF 不会走 `io::ReadImage()` / `TiffDecoder`。
	 - 反而会进入 `PrepareDngInputs()`，在 Windows 上尝试调用 Adobe DNG Converter。
	 - 这与计划文档中 “`.tif/.tiff -> TiffDecoder -> 非 RAW 管线`” 完全冲突。

 - 结论
	 - 这是当前代码的真实错误行为，而不是文档理解问题。

 - === 复核: 2026-05-23 ===
   - **已修复** — 从 `IsRawExtension()` 中移除了 `.tif` / `.tiff`（当前仅保留相机 RAW 扩展名与 `.dng`）。
   - `ClassifyInputs()` 中 TIFF 现在正确进入 `IsImageExtension()` → 归类为 `InputClass::Rgb` → 走 `TiffDecoder` + 非 RAW 管线。
   - 混合输入过滤同样连锁修复：`if (IsRawExtension(ext) || ext == ".dng")` 不再误捕 TIFF。
   - 端到端验证：TIFF 输入 → "RGB input pipeline" → 正常输出。

 ### 2. RAW 分支强制解析为 DNG 输出路径，缺失 RAW -> 非 DNG 写出分支

 - 相关文件
   - `libburstmerge/src/core/pipeline.cpp:269-273, 489-523`
   - `libburstmerge/src/core/pipeline_io.cpp:110-124`

 - 问题描述
	 - RAW 分支一开始总是调用 `ResolveOutputPath(output_path_or_dir)`。
	 - `ResolveOutputPath()` 仅识别 `.dng` 文件路径，否则生成 `burstmerge_output.dng`。
	 - 后续无论 `settings.output_format` 如何，最终仍执行 `DngWriter writer(...); writer.Write(...)`。

 - 缺失逻辑
	 - RAW + `output_format != DNG` 的 writer 分支。
	 - `Bayer` 灰阶白点归一化后写入 `PNG/JPEG/BMP/TIFF` 的流程。
	 - 对 `settings.output_format` 的 RAW 写出控制。

 - 结论
	 - 计划文档中这条路径已经规划，但代码里完全缺失。

 - === 复核: 2026-05-23 ===
   - **已修复** — 新增 RAW → non-DNG 分支（`pipeline.cpp` 中格式决策后 `else` 分支）：
     - `ConvertPlaneImageToMosaic` 后检查 `want_dng`，若非 DNG 则构造 `DecodedImage`（`kPixelGray`、Bayer mosaic）并调用 `io::WriteImage()`。
     - `output_path` 解析推迟到格式决策之后，使用统一的 `ResolveImageOutputPath()`。
   - `ResolveOutputPath()` 已整体移除（死代码清理）。
   - 端到端验证：`RAW → PNG（-o out.png）` → 正确写出文件。

 ### 3. 非 RAW 输出到 PNG/TIFF 的量化没有按 `--bit-depth` 做缩放

 - 相关文件
   - `libburstmerge/src/io/image_writer.cpp:41-53, 83-93`
   - `libburstmerge/src/io/png_writer.cpp:98-116, 122-129`
   - `libburstmerge/src/io/tiff_writer.cpp:66-80, 91-100`

 - 问题描述
	 - `WriteImage()` 将 PNG/TIFF bit depth 调整为 8 或 16。
	 - 但 writer 仅把 `image.data` clamp 到 `255` 或 `65535`。
	 - 未使用输入 `white_level`。
	 - 未将 8/10/12/14-bit 逻辑值映射到目标容器满量程。

 - 影响
	 - 例如非 RAW 8-bit 图像输出 16-bit TIFF 时，像素值往往仍停留在 `0..255`，而非扩展到 `0..65535`。

 - 结论
	 - 这与计划中“`--bit-depth` 统一管理全部输出精度”的语义不一致。
	 - 当前行为更像“容器位宽切换”，而非“数值精度输出控制”。

 - === 复核: 2026-05-23 ===
   - **已修复** — `WriteParams` 新增 `white_level` 字段；所有 writer 在量化前执行 `value / white_level * output_max`：
     - `image_writer.cpp:99-102`：计算 `wl` 为输入图像 `white_level` 最大值。
     - `png_writer.cpp:98-100`：`scale = output_max / wl`。
     - `tiff_writer.cpp:66-68`：同上。
     - `jpeg_writer.cpp:62-64`：同上（仅 8-bit）。
     - `bmp_writer.cpp:58-60`：同上（仅 8-bit）。
   - 端到端验证：8-bit JPEG 输入 → 16-bit TIFF 输出时像素值正确映射至 `0..65535`。

 ### 4. `Settings` 中存在 `dng_bit_depth` 与 `bit_depth` 两套参数，RAW / 非 RAW 路线分裂使用

 - 相关文件
   - `libburstmerge/include/burstmerge/api.h:51-56`
   - `apps/cli/main.cpp:143-149`
   - `libburstmerge/src/core/pipeline.cpp:400-442`
   - `libburstmerge/src/io/image_writer.cpp:83-91`

 - 问题描述
	 - 当前代码中：RAW 路线用 `settings_.dng_bit_depth`，非 RAW writer 路线用 `settings.bit_depth`。
	 - CLI 暂时把两者设为同一个值。

 - 风险
	 - 调库方只改 `dng_bit_depth`，非 RAW 输出不会跟着改变。
	 - 调库方只改 `bit_depth`，RAW 输出不会跟着改变。

 - 结论
	 - 文档目标是“输出精度统一参数”，但代码仍处于双通道状态，存在行为分叉风险。

 - === 复核: 2026-05-23 ===
   - **已修复** — `dng_bit_depth` 已从 `Settings` 中移除，只保留单一 `bit_depth`。
   - RAW 管线：`ResolveTargetWhiteLevel(settings_.bit_depth, ...)` 统一读取 `bit_depth`。
   - 非 RAW 管线：`WriteImage` → `AdjustBitDepth(fmt, settings_.bit_depth)` 统一读取 `bit_depth`。
   - CLI：单一 `-b / --bit-depth (8, 10, 12, 14, or 16)` → `settings.bit_depth`。
   - 不再存在"只改一个不会影响另一条路"的风险。

 ### 5. 混合输入过滤规则依赖扩展名分类，而非 decoder 真实能力

 - 相关文件
   - `libburstmerge/src/core/pipeline.cpp:123-149`

 - 问题描述
	 - 混合输入时，代码按扩展名决定是否保留 RAW：
		 - `if (ext == ".dng" || IsRawExtension(ext)) filtered.push_back(p);`
	 - `.tif` / `.tiff` 在 `IsRawExtension()` 内返回 true。

 - 影响
	 - 普通 TIFF 会被当成 RAW 保留下来，而不会像文档所说那样交给 `TiffDecoder`。
	 - 这会放大第 1 条问题，在混合输入场景下尤其明显。

 - === 复核: 2026-05-23 ===
   - **已修复** — 根因与 A.1 相同（TIFF 被误归 RAW）。`.tif` / `.tiff` 移出 `IsRawExtension()` 后，混合输入过滤自动正确：
     - TIFF 文件 → `IsRawExtension()` 返回 `false` → 不走 `filtered.push_back()` → 被跳过并报告 `"Skipping non-RAW file: ..."`。
   - 当前过滤逻辑仍依赖扩展名分类，但在 TIFF 已纠正的前提下，这是合理的：实际需要 Adobe DNG Converter 预处理的是相机 RAW 文件（`.arw` / `.cr2` / `.nef` 等），Decoder 无法直接读取它们。
   - 目前不存在已知的扩展名分类错误。

 ## B. 关键后续问题

 ### 1. RAW 显式输出非 DNG 到文件路径时，路径解析先把目标文件名当目录创建，最终写文件失败

 - 相关文件
   - `libburstmerge/src/core/pipeline.cpp:270-273, 489-517`
   - `libburstmerge/src/core/pipeline_io.cpp:110-124`

 - 问题描述
	 - RAW 分支一开始无条件执行：`output_path = ResolveOutputPath(output_path_or_dir)`。
	 - `ResolveOutputPath()` 只识别 `.dng` 为文件路径，其他带扩展名路径会被当目录处理并 `create_directories(out)`。

 - 典型场景
	 - `--output-format png -o out.png`
	 - 实际流程先创建目录 `out.png/`，返回临时 DNG 路径 `out.png/burstmerge_output.dng` 给 `PrepareDngInputs()` 使用。
	 - 后续进入 RAW -> 非 DNG 分支时，又把 `output_path` 改回 `out.png`。
	 - writer 试图写入已存在的目录路径，必然失败。

 - 结论
	 - 这是当前最严重的问题，因为它直接让 RAW -> 非 DNG 文件输出在常见用法下不可用。

 - === 复核: 2026-05-23 ===
   - **已修复** — 见 A.2 的修复。根因是 `ResolveOutputPath()` 提前执行 + 只认 `.dng` 为文件。修复措施：
     1. 移除提前的 `ResolveOutputPath()` 调用，`PrepareDngInputs` 只传原始 `output_path_or_dir` 作为转换上下文（不影响最终路径）。
     2. 最终 `output_path` 在格式决策之后用 `ResolveImageOutputPath()` 统一解析（所有扩展名均视为文件）。
     3. `ResolveOutputPath()` 已删除，不再存在"非 DNG 路径当目录创建"的行为。
   - 端到端验证：`RAW --output-format png -o out.png` → 正确写出 `out.png` 文件，不创建目录。

 ### 2. RGB 路线默认输出名仍然写死为 `burstmerge_output.png`

 - 相关文件
   - `libburstmerge/src/core/pipeline.cpp:229-256, 258`

 - 问题描述
	 - RGB 分支的默认输出名逻辑仍写死为：
		 - 空输出 -> `burstmerge_output.png`
		 - 输出是目录 -> `<dir>/burstmerge_output.png`
	 - 真正写出格式由 `io::WriteImage(..., settings_)` 决定。

 - 影响
	 - `--output-format jpeg -o outdir` 会得到扩展名 `.png`、内容却是 JPEG 的文件。
	 - `--output-format tiff / bmp` 也会出现同样问题。

 - 结论
	 - 这会造成错误的文件扩展名，影响打开方式、脚本处理和用户认知。
	 - RAW 非 DNG 路线已经做了按格式选默认扩展名，RGB 路线明显没有同步。

 - === 复核: 2026-05-23 ===
   - **已修复** — RGB 分支改用 `ResolveImageOutputPath(output_path_or_dir, settings_.output_format)`：
     - 目录输出：`<dir>/burstmerge_output.<fmt_ext>`（如 `--output-format jpeg` → `.jpg`）。
     - 空输出：`burstmerge_output.<fmt_ext>`（同上）。
     - 显式文件路径：扩展名不匹配时自动校正（如 `-o out.png --output-format jpeg` → 写 `out.jpg`）。
   - 端到端验证：`JPEG --output-format jpeg -o outdir` → 产生 `outdir/burstmerge_output.jpg`。

 ## E. 其他备注

 1. `include/.../image_decoder.h` 计划中指定 `using PixelFormat = uint32_t;` 作为类型别名，但实际代码使用裸 `constexpr uint32_t`。
    - 这是为了规避与 `image_buffer.h` 中现有 `enum class PixelFormat` 的命名冲突。
    - 这是可接受的调整，但值得记录。

    - === 复核: 2026-05-23 ===
      - 维持原判断：可接受调整，不修。

 2. `src/io/png_decoder.cpp:102-105` 可能造成精度丢失：当输入位深为 16 时，`png_set_scale_16(png)` 会缩放至 8 位。
    - 计划中写明 PNG 解码器应支持 8 和 16 位。
    - 应使用 `png_set_strip_16`（或保留 16 位并相应设置 `bit_depth = 16`）。
    - 当前行为对 `--bit-depth 16` 管线会产生静默裁剪。

    - === 复核: 2026-05-23 ===
      - **已修复** — `png_set_scale_16` 已移除，改用 `png_set_swap`（字节序交换）：
        - 16-bit 输入读取为 `uint16_t` 行，转为 float 时保留 `0..65535` 范围。
        - `white_level` 设为 `65535.0f`，`bit_depth` 设为 `16`。

 3. `src/io/png_decoder.cpp:110` `png_set_gray_to_rgb(png)` 强制将灰度 PNG 转换为 RGB。
    - 计划中要求灰度应保留为 `kPixelGray`。
    - 这不符合灰度输入的要求。

    - === 复核: 2026-05-23 ===
      - **已修复** — `png_set_gray_to_rgb` 已移除。GRAY 保留单通道 → `kPixelGray`；GRAY_ALPHA 用 `png_set_strip_alpha` 去 alpha 后同样保留为 `kPixelGray`。

 4. `src/io/png_decoder.cpp:104, 122` 硬编码 `bit_depth = 8`（在缩放后），无法正确传递 16 位元数据。
    - 如果修复了 `png_set_scale_16`，`info.bit_depth` 应反映原始位深。

    - === 复核: 2026-05-23 ===
      - **已修复** — `bit_depth` 改为 `(原始 bit_depth == 16) ? 16 : 8`，`white_level` 相应设为 `65535.0f` 或 `255.0f`。

 5. `src/core/pipeline.cpp:236, 244` RGB 输出路径硬编码 `.png` 扩展名作为默认值。
    - 如果在输出目录未指定时通过 `WriteImage` 使用 `OutputFormat::Auto -> PNG`，这种写法是可行的。
    - 但如果将来支持其他格式，可能会造成混淆。

    - === 复核: 2026-05-23 ===
      - **已修复** — RGB 分支改用 `ResolveImageOutputPath(output_path_or_dir, settings_.output_format)`。
      - 进一步在函数内增加了扩展名自动校正（扩展名与格式不匹配时静默替换为规范扩展名）。
      - 三条管线共享同一函数，不再有硬编码 `.png` 的路径。

 6. `src/io/dng_writer_adapter.cpp` DNG writer 适配器仍是一个会抛出异常的桩模块（"use the dedicated RAW pipeline"）。
    - 计划中写明“DngWriter (改造) — 适配 FloatImage 输入”。
    - 桩模块避免了错误的调用路径，但并未真正适配 `FloatImage` 输入。
    - 实际上 DNG 写入确实需要完整的 RAW 元数据上下文，因此当前架构决策是合理。

    - === 复核: 2026-05-23 ===
      - 维持原判断：DNG 写入需要 RAW 元数据上下文，非 RAW 管线调用 DNG writer 是错误用法，桩模块正确防止了误用。当前架构决策仍合理。

## 补充: 当前代码中未彻底修好的相邻问题

虽然本文件列出的大多数 2026-05-22 问题确实已经修复，但当前代码里仍有若干与这些主题相邻、且会影响“修复彻底性”判断的新问题：

### 复核: 2026-05-23

以下逐条评估上述 4 条"未彻底修好的相邻问题"在 **2026-05-23 修改后**的状态：

**1. `bit_depth 8/10` RAW 白电平**
- 当前状态：**已修复**
- 复核依据：`libburstmerge/src/core/pipeline.cpp:37-53`
- `ResolveTargetWhiteLevel` 使用 `(1u << bit_depth) - 1u` 统一计算，`bit_depth=8` → 255，`bit_depth=10` → 1023，表述准确。
- 注意：未使用 switch-case 硬编码表，而是用移位表达式，行为等价且更简洁。边界检查（`<=0 → sensor_white`、`>16 → sensor_white`）也已对齐。

**2. `auto` 输出格式与显式扩展名脱节**
- 当前状态：**已修复**
- 复核依据：`libburstmerge/src/core/pipeline.cpp:303-325`; `libburstmerge/src/io/image_writer.cpp:100-104`
- RGB 管线：`Auto` 先 `InferFormatFromExtension(output_path_or_dir, fallback)` 尝试从文件名扩展名推断格式，匹配失败才 `InferOutputFormat` 兜底 → 两条管线行为对称。
- `WriteImage()`：使用 `InferFormatFromExtension(path, InferOutputFormat(settings, all_raw))` 确保实际 writer 与路径扩展名一致，`-o out.jpg --output-format auto` 不会再写出 PNG 内容。
- 修复链与 `docs/rgb-pic-plan-dsv4.md` 中"`output_format` 优先级"新节一致。

**3. mixed 输入未知扩展名静默跳过**
- 当前状态：**已修复（按契约报错）**
- 复核依据：`libburstmerge/src/core/pipeline.cpp:206-209`
- 混合输入过滤的条件已改为：
  ```
  if (IsRawExtension(ext) || ext == ".dng" || !IsImageExtension(ext))
  ```
  未知扩展名（`!IsImageExtension`）被保留至 RAW 管线，随后在 `PrepareDngInputs` 中遇到不可转换文件时抛出 `std::runtime_error`（见注释 _"they will fail in PrepareDngInputs with a clear error, rather than being silently dropped"_）。
- 行为符合 `docs/plan-dsv4.md` 对 mixed 输入的契约："fail with clear error" 而非静默丢弃。

**4. `TiffDecoder` 32-bit 非 float 误读为 float**
- 当前状态：**已修复**
- 复核依据：`libburstmerge/src/io/tiff_decoder.cpp:53-92`
- float 路径严格限于 `SAMPLEFORMAT_IEEEFP`（`is_float`），`bps == 32` 不再单独触发 float 模式。
- 新增专用 `else if (bps == 32)` 分支：`std::vector<uint32_t>` 直接读取原始 uint32 位模式，然后以 `static_cast<float>` 安全转换为浮点值，避免了对随意位模式进行 float-interpret 的风险。
- 数据解释不再存在歧义。

**5. BMP decoder 边界校验偏弱**
- 当前状态：**已修复**
- 复核依据：`libburstmerge/src/io/bmp_decoder.cpp:53-120`
- 新增以下防御性校验：
  - `biSize < 40` → 拒绝（旧 DIB 头缺少 `biClrUsed` 等字段）。
  - `bfOffBits < dib_end` → 拒绝（调色板位于 DIB 头内）。
  - 调色板定位基于 `sizeof(BmpFileHeader) + ih.biSize`（兼容 BITMAPV4HEADER/V5HEADER 等变体）。
  - 使用 `biClrUsed`（非零时）替代硬编码 256 确定调色板条目数。
  - 调色板 `gcount()` 校验防截断。
  - `bfOffBits` 必须越过调色板尾部。
- 边界校验已覆盖常见的错误 / 畸形 BMP 文件场景。
