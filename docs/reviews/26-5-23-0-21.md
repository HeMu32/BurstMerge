### 1. `--bit-depth 8` / `10` 未落地到 RAW/DNG 白电平目标

- 相关文件
    - `libburstmerge/src/core/pipeline.cpp:37-45, 427-434`
    - `apps/cli/main.cpp:103-145`
    - `libburstmerge/include/burstmerge/api.h:51-54`

- 问题描述
    - CLI 和 Settings 明确允许 `8/10/12/14/16`。
    - 但 `ResolveTargetWhiteLevel()` 只处理了 `12/14/16`。
    - 结果是 `8` 和 `10` 会静默退回到 `sensor_white`。

- 影响
    - RAW -> DNG 输出时，`8` 不会变成 `255`。
    - RAW -> DNG 输出时，`10` 不会变成 `1023`。
    - RAW -> 非 DNG 路线中，`target_white` 也拿不到 8/10-bit 的目标值。

- 结论
    - 这不是文档偏差，而是实际行为与公开接口不一致。
    - 用户传了合法参数，却得不到对应位深语义。
    - 这是输出位深接口语义未真正落地的问题。

- === 修补: 2026-05-23 ===
    - **已修复** — `pipeline.cpp:ResolveTargetWhiteLevel()` 添加了 `case 8: return 255; case 10: return 1023;`。
    - RAW -> DNG 与 RAW -> 非 DNG 两条路径现在都能得到正确的 8/10-bit `target_white`。
    - 8-bit / 10-bit 现在正确映射到 `255` / `1023` 白电平目标。

### 2. `--output-format auto` + 显式扩展名可能导致输出内容与扩展名不一致

- 相关文件
    - `libburstmerge/src/core/pipeline.cpp:96-149, 284-286`
    - `libburstmerge/src/io/image_writer.cpp:30-39, 70-104`

- 问题描述
    - `ResolveImageOutputPath()` 只有在 `fmt != Auto` 时才会纠正扩展名。
    - RGB 路线先用 `settings_.output_format` 解析路径，再由 `WriteImage()` 后续推断真实输出格式。

- 典型场景
    - `--output-format auto -o out.jpg` + 全部输入为 PNG/JPEG/BMP/TIFF。
    - `ResolveImageOutputPath(..., Auto)` 保留 `.jpg`。
    - `WriteImage()` 推断真实格式为 PNG。
    - 最终生成的是一个 PNG 内容、`.jpg` 扩展名的文件。

- 进一步说明
    - 同类问题也会出现在 `out.dng`、`out.tif` 这类显式路径上。
    - 问题当前集中于 `Auto + 显式扩展名` 的组合，但很容易踩中。

- === 修补: 2026-05-23 ===
    - **已修复** — 见 C.3 修补。管线现在在 Auto 时先尝试 `InferFormatFromExtension()` 从文件名推断格式，
      再以推断结果传入 `ResolveImageOutputPath()`，路径解析与写出格式不再脱节。
    - `WriteImage()` 内部同样改用 `InferFormatFromExtension()`，确保实际写入的内容与扩展名一致。
    - 扩展名不匹配时静默修正，并通过 `Report()` 警告用户实际使用的格式。

### 3. mixed 输入分支中“未知扩展名视作 RAW，稍后报 unsupported”契约不成立

- 相关文件
    - `libburstmerge/src/core/pipeline.cpp:75-94, 180-203`

- 问题描述
    - `ClassifyInputs()` 的注释和行为认为：未知扩展名 -> 当成 RAW，后续再报错。
    - 但 mixed 分支实际处理时，未知扩展名文件被静默跳过。

- 后果
    - 如果输入中同时存在一个已知 RGB 文件和一个未知扩展名文件：
        - 先判成 Mixed。
        - 再把未知扩展名文件当“non-RAW”跳过。
        - 静默继续处理剩下的 RGB 文件。
    - 这会吞掉用户错误输入，尤其危险于批处理场景。

- === 修补: 2026-05-23 ===
    - **已修复** — 见 C.4 修补。mixed 分支现在保留未知扩展名文件：
        - 过滤条件从 `IsRawExtension(ext) || ext == ".dng"` 扩展为
          `IsRawExtension(ext) || ext == ".dng" || !IsImageExtension(ext)`。
        - 未知扩展名文件不会再被静默丢弃，而会进入 RAW 准备阶段并报出清晰错误。
    - 现在实现与 `ClassifyInputs()` 的契约一致。

### 1. `png_decoder.cpp:96-107` — GRAY + tRNS 分支导致通道数不匹配

- 场景
    - PNG 为 `PNG_COLOR_TYPE_GRAY` 且包含 `tRNS` 透明块时。

- 问题
    - `png_set_tRNS_to_alpha(png)` 会添加 Alpha 通道。
    - 但 `color_type == GRAY_ALPHA` 判断仍为 `FALSE`（原始为 GRAY）。

- 风险
    - `DecodedImageToFloatImage()` 会按 `width*height*4` 地址访问，必然越界。

- 建议修复
    - 对所有非 RGB 输入无条件剥离 Alpha 并警告。

- === 修补: 2026-05-23 ===
    - **已修复** — `png_decoder.cpp` 现在对 `GRAY` 和 `GRAY_ALPHA` 两类输入都执行 `png_set_strip_alpha(png)`。
    - 这同时覆盖了直接 `GRAY_ALPHA` 和 `GRAY + tRNS` 两条路径，保证灰度 PNG 最终始终为 1 通道。
    - `png_get_channels()` 与 `pix_fmt` 现在保持一致，不再存在越界风险。

### 1. `TiffDecoder` 的 `SAMPLEFORMAT` / 32-bit 读取逻辑不安全

- 相关文件
    - `libburstmerge/src/io/tiff_decoder.cpp:41-49, 59, 65-68, 72-90`

- 问题
    - `sample_format` 仅声明后传给 `TIFFGetField(...)`，没有默认值。
    - `TIFFTAG_SAMPLEFORMAT` 不是必选标签，缺失时 `sample_format` 保持未初始化。
    - 后续分支使用 `if (is_float || bps == 32)`，只要 `BitsPerSample == 32` 就分配 `std::vector<float>` 并把 `TIFFReadScanline` 读入 float buffer。

- 结果
    - 普通 `uint32` TIFF 会被当作 float bit pattern 解释，像素值完全错误。
    - 缺失 `SAMPLEFORMAT` 的 TIFF 行为不确定，甚至可能在不同机器上随机进入 `is_float` 路径。

- 结论
    - 这是实打实的数据损坏问题。

- === 修补: 2026-05-23 ===
    - **已修复** — `tiff_decoder.cpp`：
        - `sample_format` 初始化为 `SAMPLEFORMAT_UINT`（1），`TIFFGetField` 失败时不改值，unsigned int 为最安全的默认假设。
        - `white_level` 计算使用 `(1ull << bps)` 避免 UB。
        - `is_float` 仅在 `TIFFTAG_SAMPLEFORMAT` 明确为 `IEEEFP` 时为 true。无该标签的 32-bit TIFF 不会误入 float 路径。
    - **修补不足**：`if (is_float || bps == 32)` 仍在 float 路径中保留 `bps == 32`，所有 32-bit TIFF（含 SAMPLEFORMAT_UINT）仍走 float 读取，uint32 bit pattern 会被误读为 float 值。

- === 复核: 2026-05-23 ===
    - **追加修复** — 将 `if (is_float || bps == 32)` 改为纯 `if (is_float)`，浮点路径只走 IEEEFP。
    - 新增独立的 `else if (bps == 32)` 分支，使用 `std::vector<uint32_t>` 读取后 `static_cast<float>` 转换。
    - 现在 SAMPLEFORMAT_UINT 的 uint32 TIFF 不再被 float 路径误读。

### 2. BMP 8-bit 读写路径不符合常规 BMP 语义

- 相关文件
    - `libburstmerge/src/io/bmp_writer.cpp:61-70, 92-118`
    - `libburstmerge/src/io/bmp_decoder.cpp:79-82, 91-123`

- 问题
    - `BmpWriter` 对 8-bit 灰度 BMP 只写文件头 + info header + 索引字节，没有写 256 项灰度 palette，也没有把 `bfOffBits` 指向 palette 后面。
    - `BmpDecoder` 对 8-bit BMP 直接把像素字节当灰度值读取，忽略 palette / color table。

- 后果
    - 自己写出的 8-bit BMP 可能被标准查看器错误显示或拒绝。
    - 读取外部 8-bit paletted BMP 时，得到的是 palette index，不是真正灰度值。
    - 文档中说支持 BMP，但当前 8-bit BMP 实际上是一种非常狭窄、非标准的私有变体。

- === 修补: 2026-05-23 ===
    - **已修复** — Writer 新增 256 项灰度 palette 写入并修正 `bfOffBits`；Decoder 新增 palette 读取与索引映射。
    - 写出和读取的 8-bit BMP 现在符合标准 BMP 规范。
    - **修补不足**：Decoder 硬编码 `sizeof(BmpFileHeader) + sizeof(BmpInfoHeader)` 作为 palette 位置，未校验 `biSize`（DIB 头可能有更大的变体）、`biClrUsed`（实际调色板条目数）、`bfOffBits`（是否越过 palette 尾部）。

- === 复核: 2026-05-23 ===
    - **追加修复** — `bmp_decoder.cpp` 新增以下防御性校验：
        - `ih.biSize < 40` → 拒绝（需要 BITMAPINFOHEADER 以上版本以存在 biClrUsed）。
        - `fh.bfOffBits < sizeof(BmpFileHeader) + ih.biSize` → 拒绝（bfOffBits 位于 DIB 头内部）。
        - 使用 `biClrUsed`（非零时）代替硬编码 256，`std::min(biClrUsed, 256u)` 钳位上限。
        - 按 `sizeof(BmpFileHeader) + ih.biSize` 定位 palette，兼容更长的 DIB 变体。
        - palette 读取后校验 `gcount()` 防截断。
        - `fh.bfOffBits` 必须越过 palette 尾部。
    - 不再假设 `sizeof(BmpInfoHeader)` == biSize。

### 3. `output-format=auto` 仍然存在路径解析与实际写出格式分离的问题

- 相关文件
    - `libburstmerge/src/core/pipeline.cpp:102-149, 284-286`
    - `libburstmerge/src/io/image_writer.cpp:30-39, 70-104`

- 问题
    - `ResolveImageOutputPath()` 只在 `fmt != Auto` 时纠正扩展名。
    - RGB 路线先根据 `settings_.output_format` 解析路径，再由 `WriteImage()` 使用 `InferFormat(...)` 推断真实输出格式。

- 结果
    - `--output-format auto -o out.jpg` + 全非 RAW 输入，会生成 `.jpg` 扩展名但实际 PNG 内容的文件。

- 结论
    - 这是 auto + 显式输出文件名的直接一致性漏洞。

- === 修补: 2026-05-23 ===
    - **已修复** — `image_writer.h/.cpp`：
        - 新增 `OutputFormatToString()`（公开辅助）。
        - 新增 `InferFormatFromExtension(path, fallback)`，从输出文件名扩展名推断格式（`.png`/`.jpg`/`.jpeg`/`.bmp`/`.tif`/`.tiff`/`.dng`）。
        - 原有 `InferOutputFormat()` 保留作为纯 fallback 逻辑（不可用于 Auto 时的路径推断）。
    - `pipeline.cpp` 两个分支（RGB + RAW）：
        - 当 `settings_.output_format != Auto` 时直接使用（不警告）。
        - 当 `settings_.output_format == Auto` 时，优先 `InferFormatFromExtension()`。
            - 若从扩展名推断到有效格式 → `Report()` 警告 "inferred {fmt} from filename extension"。
            - 若扩展名无匹配 → 回退到 `InferOutputFormat()` 的 fallback → `Report()` 警告 "defaulting to {fmt}"。
    - `image_writer.cpp:WriteImage` 中也改用 `InferFormatFromExtension(path, ...)` 取代原先的 `InferOutputFormat(settings, ...)`，
      确保实际写出的格式与管道推断结果一致，不会出现扩展名告诉用户 JPEG 但实际写入 PNG 的情况。
    - 路径解析与写出的格式推断现在共享同一份 extension 感知逻辑，不会分离。

### 5. `JpegDecoder` 将所有非 1/3 通道 JPEG 当成 RGBA 处理

- 相关文件
    - `libburstmerge/src/io/jpeg_decoder.cpp:71-83, 85-99`

- 问题
    - 代码如下：

- === 修补: 2026-05-23 ===
    - **已修复** — `jpeg_decoder.cpp`：
        - 检查时机从 `jpeg_start_decompress` 之后提前到 `jpeg_read_header` 之后。
        - 检查 `cinfo.num_components`（原始编码分量数），而非 `output_components`。
        - 通过 `cinfo.jpeg_color_space` 获取实际色彩空间名称（CMYK/YCCK/YCbCr···），构造详细错误消息。
        - 错误消息示例：
          ```
          JpegDecoder: unsupported color space CMYK (4 components).
          Only 1-channel (grayscale) and 3-channel (RGB) JPEG are supported.
          Use an external tool to convert to sRGB first.
          ```
        - 不内置 CCM 转换，留给外部工具处理。
